## 1. 产品概述

### 1.1 产品定位

**临床病人管理助手** 是一款面向临床医生的移动端 PWA 应用，用于管理住院病人的查房信息、待办事项和医嘱记录。

**核心价值**：
- 📱 **移动优先** - 手机上使用，随时随地记录
- ⚡ **秒开** - 本地存储，无网络也能用
- 🤖 **智能** - 自动解析待办时间，智能提醒
- 📋 **高效** - 批量导入病人，快捷操作

### 1.2 产品边界

**包含**：
- ✅ 病人信息管理（增删改查）
- ✅ 待办事项管理（智能时间解析 + 提醒）
- ✅ 查房列表（自定义排序）
- ✅ 批量导入/导出
- ✅ 快捷操作
- ✅ 每日小结

**不包含**（进阶功能，未来考虑）：
- ❌ 多用户协作
- ❌ 统计分析/报表
- ❌ 云端同步
- ❌ 原生 App

### 1.3 设计原则

1. **效率优先** - 每次操作不超过 3 步，查房场景下可单手完成
2. **容错设计** - 所有删除操作可撤销，导入前可预览
3. **离线可用** - 不依赖网络，任何场景都能使用
4. **信息密度** - 一页显示 40+ 病人摘要，关键信息一目了然

### 1.4 核心术语

| 术语 | 解释 |
|------|------|
| 查房 | 医生按固定顺序巡视病人 |
| 待办 | 需要完成的任务，含时间解析后的目标日期 |
| 提醒 | 基于目标日期生成的到期/逾期提示，以及根据换药频率/查血日计算的需换药、需查血提示 |
| 分组 | 病区的医疗组（如解组、勇组），带颜色标签 |
| 快捷待办 | 设置中自定义的常用待办模板 |
| 每日小结 | 当天完成的待办和换药/查血记录汇总 |

## 2. 用户画像

### 2.1 目标用户

| 属性 | 描述 |
|------|------|
| **角色** | 临床医生（住院医师/主治医师） |
| **科室** | 骨科（可扩展到其他外科科室） |
| **技术水平** | 熟练使用智能手机 |
| **使用场景** | 查房时、办公室、家中 |

### 2.2 用户痛点

1. **病人多，容易漏** - 40+ 病人，每天哪些要查血、哪些要换药，容易忘记
2. **待办分散** - 每个病人的待办不同，难以统一管理
3. **时间感知弱** - "星期二出院"这种待办，到了当天容易忘记
4. **数据录入慢** - 每天要重新输入病人信息，效率低

### 2.3 用户目标

1. **快速记录** - 查房时快速记录每个病人的待办
2. **智能提醒** - 系统自动提醒今天要做的事
3. **高效导入** - 粘贴最短文本，快速同步全量数据
4. **每日小结** - 快速生成当天完成的事项，用于交班/记录

### 2.4 使用场景矩阵

| 场景 | 时间 | 设备 | 目标 | 关键功能 |
|------|------|------|------|----------|
| 早晨查房 | 08:00-10:00 | 手机 | 查看今日提醒并按顺序查房 | 提醒栏、查房列表 |
| 查房记录 | 10:00-11:00 | 手机 | 快速添加/完成待办 | 快捷待办、病人详情 |
| 办公室整理 | 14:00-15:00 | 桌面/平板 | 整理待办、调整顺序 | 待办页、设置 |
| 交接班 | 17:00-18:00 | 任意 | 生成今日小结 | 每日小结 |
| 每日更新名单 | 查房前（每日） | 手机/桌面 | 更新当天病人列表（粘贴最新 roster，按姓名匹配增改） | 批量导入 |

## 3. 核心工作流

### 3.1 每日工作流

```
早晨查房前
  ↓
更新当天病人列表
（粘贴最新 roster，按姓名匹配增改）
  ↓
打开 App，查看今日提醒
（查血、换药、待办到期）
  ↓
查房（按自定义顺序）
  ↓
给每个病人添加/更新待办
（系统自动解析时间，到期提醒）
  ↓
查房结束，生成每日小结
（复制给上级/交班）
  ↓
随时查看待办，完成打勾
```

### 3.2 备份工作流

```
每日查房前（详见 3.1）
  ↓
更新当天病人列表（粘贴 roster，按姓名匹配增改）
  ↓
每天查房，添加/更新待办
  ↓
（按需，建议每周）导出数据备份
（防止丢失）
```

### 3.3 病人全生命周期

```
更新名单（批量导入）/ 手动添加
  ↓
维护基本信息（床号、诊断、分组、手术日期）
  ↓
查房过程中添加/完成待办
  ↓
系统根据换药频率/查血日自动提醒
  ↓
病人出院或转科 → 手动删除
```

### 3.4 待办全生命周期

```
输入待办文本
  ↓
实时时间解析（今明后天/星期几/具体日期）
  ↓
保存到病人名下
  ↓
首页/待办页显示剩余天数
  ↓
到期当天高亮提醒
  ↓
完成后打勾
  ↓
每日小结统计已完成项
```

## 4. 功能需求

### 4.1 病人管理

#### 4.1.1 病人列表
**功能描述**：
- 显示所有病人，按**自定义查房顺序**（`Settings.roundingOrder`）排序。
- 支持**分组筛选**（如：解组、勇组）。
- **组合卡片（grouped card）展示（v2.6 新增，呼应 4.9.3）**：属于查房序列中**同一病房块（room block）**的病人，以**一张组合卡片**聚合展示——组合卡片代表一个病房块，卡内罗列该房所有病人（按基础床号升序）；点击组合卡片可展开/收起，点击单个病人进入 `/patient/{id}` 详情。未归入任何病房块的病人仍以单人卡片展示。
- 每个病人卡片显示：
  - 左侧床号色块（取床号末两位数字或前 3 位，背景为分组颜色）。
  - 中部姓名、诊断。
  - 分组名称标签（带颜色）。
  - 未完成待办数量。
  - 逾期/今日到期/需换药/需查血状态徽章。

**排序规则**：
1. 按 `roundingOrder`（查房序列，见 4.9.3）的单元顺序排序；病房块内按基础床号升序。
2. 未进入查房序列的病人，按「病区 + 基础床号数值」升序追加到末尾（病区与基础床号由 4.9.2 识别得出，如 `309W23` → 病区 309W、床号 23）。
3. 启用分组筛选时，列表只显示该分组病人，仍按上述规则排序。
4. 同一病人不重复显示。

**交互细节**：
- 点击卡片 → 进入 `/patient/{id}` 详情页。
- 长按卡片（移动端）或点击右侧 `...` → 显示操作菜单（更改分组、删除）。

**字段说明**（病人卡片中展示/参与提醒）：
- 床号：必填，用于排序和识别。
- 姓名：必填，用于批量导入匹配。
- 诊断：必填。
- 分组：可选，带颜色标签。
- 手术日期：可选，YYYY-MM-DD。
- 换药频率：可选，1-90 整数天。
- 上次换药日期：可选，YYYY-MM-DD。
- 查血日期：可选，可填写多个周几名称（如“周一 周三”）。

**验收标准**：
- [ ] 列表加载时间 < 500ms（40 个病人）。

#### 4.1.2 添加/编辑病人
**功能描述**：
- 在 Bottom Sheet 中手动添加或编辑病人信息。
- 字段：床号、姓名、诊断、分组名称、分组颜色、手术日期、换药频率（天）、上次换药日期、查血日期。

**交互细节**：
- 分组输入提供 `解组、勇组、李组、王组` 预设选项。
- 分组颜色使用原生 `<input type="color">`，默认取预设颜色列表。
- 保存时校验床号、姓名、诊断是否为空，否则提示。
- 新增病人时**不**自动加入 `roundingOrder`，未在顺序中的病人按床号自然顺序排在后面。
- 编辑床号后**不**同步修改 `roundingOrder`，需要用户手动调整。
- 编辑状态下提供“删除病人”按钮，删除后同步删除该病人的所有待办。

**表单校验**：
- 床号、姓名、诊断为必填。
- 换药频率（若填写）应为整数。
- 日期格式使用原生日期选择器，保证格式正确。

**验收标准**：
- [ ] 保存后立即更新列表。
- [ ] 分组颜色实时更新。
- [ ] 支持删除病人并级联删除待办。

#### 4.1.3 批量导入病人
**功能描述**：
- 在首页 Header「添加病人」中进入“批量导入”（Sheet 内「批量导入」tab），粘贴文本后预览并导入。
- 文本格式：`床号 姓名 诊断`（如 `309W23 魏亿鑫 取除骨折内固定装置`）。
- 导入逻辑：
  - 按**姓名**精确匹配现有病人，匹配到则更新床号、诊断。
  - 未匹配到则新增，默认分组为空、分组颜色 `#e2e8f0`。
  - 导入文本中未出现的病人，默认**保留**；仅在导入预览中显式勾选“移除未出现病人”时才归入“删除”，确认后删除并级联删除其待办。预览中展示将被删除病人的未完成待办数量。

**交互细节**：
- 导入前显示预览：新增 / 更新 / 删除的人数及明细列表。
- 若用户勾选“移除未出现病人”且存在待删除病人，弹出二次确认对话框，并显示各病人的未完成待办数量。
- 导入后 Toast 提示：新增 X 人、更新 X 人、删除 X 人。
- 文本框始终保留，支持修改后重新预览。

**文本解析规则**：
- 每行一个病人。
- 字段以空白字符（空格或制表符）分隔，至少 3 列：床号、姓名、诊断。
- 第三列及以后合并为诊断。
- 忽略空行与前后空白。
- 行中缺少姓名或床号时跳过，并在预览中列出被跳过行号与原因。
- 姓名匹配大小写敏感。

**验收标准**：
- [ ] 导入 40 个病人 < 3 秒。
- [ ] 同名病人正确更新，不重复创建。
- [ ] 删除确认弹窗清晰明了。

### 4.2 待办管理

#### 4.2.1 添加待办
**功能描述**：
- 每个病人可以有多个待办。
- 待办内容：自由文本（如“星期二出院”、“后天换药”、“周四复查X光”）。
- 待办类型：换药、查血、开术前、明天出院、康复会诊、会诊、复查、开查血、其他。
- 系统基于 `lib/time-parser` 自动解析文本中的时间。

**智能时间解析**：
- 支持识别：
  - 今明后天：今天、明天、后天（注意：暂不支持“大后天”）。
  - 星期：周日~周六、星期日到星期六、周X、下星期X、下礼拜X、下下X。
  - 具体日期：YYYY-MM-DD、YYYY/MM/DD、MM/DD、MM-DD、M月D日。
- 解析优先级：今明后天 > 星期 > 具体日期。
- 解析成功后计算 `dueDate` 和剩余天数。

**实现细节**：
- 星期解析规则：以当前日期为基准，计算目标星期与今日星期差值；若文本包含“下星期/下礼拜/下下”且差值为 0，则返回 7 天后（下周一）而非今天。
- 当文本同时包含多种时间表达时，取优先级最高的第一个结果（如“明天星期二”会识别为“明天”）。
- 具体日期省略年份时，默认使用当前年份。
- 解析成功返回 `{ date: 'YYYY-MM-DD', label: '今明后天' | '星期几' | '具体日期' }`，失败返回 `null`。
- 实时解析通过 `useEffect` 监听输入变化，延迟由 React 渲染周期决定，通常 < 100ms。

**交互细节**：
- 添加待办使用 Bottom Sheet 中的表单。
- 内容输入框为 textarea，输入时实时调用 `parseTime` 显示解析结果。
- 解析成功时显示：`识别到时间：{date}（{label}）`。
- 解析失败时显示：`未识别到时间`。
- 提供类型下拉选择框（默认值“其他”）。
- 快捷待办按钮点击时，自动以该快捷待办的内容和类型创建待办。

**验收标准**：
- [ ] 时间解析准确率 > 90%。
- [ ] 解析结果实时显示（< 300ms）。
- [ ] 支持三种格式。

#### 4.2.2 待办列表
**功能描述**：
- 待办页显示所有病人的待办，支持分段筛选：
  - 全部 / 未完成 / 已完成 / 今天到期 / 已逾期。
- 待办页顶部新增分段切换：**病人待办 / 通用待办**。
  - 「病人待办」：现有全部病人待办，沿用上述筛选。
  - 「通用待办」：仅显示 `patientId` 为空的通用待办（见 4.8）。
- 点击病人待办进入对应病人详情页；通用待办点击可查看/完成/删除，不跳转病人页。

**排序规则**：
1. 未完成优先，逾期/今日到期待办权重更高。
2. 按 `dueDate` 升序（先到期的在前）。
3. 无日期的待办排在最后。
4. 同一天内按 `createdAt` 倒序（最新创建的在前）。

**交互细节**：
- 每个待办显示：
  - 病人床号与姓名（待办页）。
  - 待办内容。
  - 剩余天数标签（还剩 X 天 / 今天到期 / 已逾期 X 天）。
  - 状态复选框。
  - 类型徽章（换药、查血等）。
- 左滑待办项露出右侧“完成”与“删除”按钮。
  - 左滑水平位移超过 50px 时展开操作区，露出宽度 7rem。
  - 右滑超过 50px 或点击内容区域收起。
  - 操作按钮为圆形按钮：完成（绿色 success）、删除（红色 danger）。
- 点击复选框直接切换完成状态。
- 已完成待办显示删除线与 60% 透明度，数据保留。

**验收标准**：
- [ ] 待办列表加载 < 500ms。
- [ ] 到期待办高亮显示。
- [ ] 完成待办后列表实时更新。

#### 4.2.3 智能提醒
**功能描述**：
- 每次打开应用时检查待办、换药、查血状态。
- 在首页顶部显示提醒栏，汇总今日到期、已逾期、需换药、需查血的数量。

**提醒优先级**：

### 4.9 查房顺序与床号管理

> 本模块集中定义「床号如何解析」与「查房序列如何生成」两大问题。床号是医院层级里的最小单位，真实编码常复合了「病区—病房—病床（含各类型）」多层信息；而查房是按「病房（房间）」逐房展开的。本模块提供一套**可配置、通用**的识别方案，以及内置示例排序，使系统既能开箱即用，也能适配不同医院的床号编码。
> 配置入口位于设置页的「床号识别」与「查房顺序」项（见 4.6.2、4.6.3）；本模块为完整功能定义。

#### 4.9.1 功能概述

- **为什么需要**：基础床号本应是 `1/2/3…` 这样的正整数顺序（各病区实际床位数不同，**无固定上限**），但医院里因不同病区、不同类型病床（普通床、抢救床、隔离床、加床等），出现了复杂编码来定位「这张床在医院的哪个位置」。系统必须能从中识别出「病区 / 病房 / 病床」三层，才能正确排序与展示。
- **两个核心步骤**：
  - **Step 1（识别）**：从导入的床号文本中解析出「病区、病房、病床」，并界定每张床是真实床 / 真实加床 / 虚拟床（见 4.9.2）。
  - **Step 2（展示）**：针对真实床，按「病房在前、房内床在后」组织查房序列，并提供内置示例与自定义拖拽（见 4.9.3）。
- **与每日工作流的关系**：每日查房前更新病人名单（见 3.1），导入即触发 Step 1 识别；查房序列（Step 2）决定病人列表（4.1.1）的展示顺序。

#### 4.9.2 床号结构模型与通用识别（Step 1：找到所有真实床）

**床号结构模型**（以默认示例 `309W01` / `309WJ01` / `309E01` / `309WYZ01` 为例）：
```
[病区基底 3位数字][病区方位 1位字母][特殊床标记 0~2位字母][基础床号 2位数字]
    309              W              (空=真实床 / J / YZ)        01
```
- **病区（Ward）** = 病区基底 + 病区方位，如 `309W`、`309E`。`309` 代表楼栋/楼层，`W/E` 代表西/东病区。
- **特殊床标记**：可选。`J` = 病房额外床（加床），由「真实床 + 虚拟床」组成——真实床有物理位置（查房时走访），虚拟床无物理位置（不走访）；`YZ` = 另一种形式的虚拟床（无物理位置，不走访）。无标记者为基础**真实床**（房内床）。
- **基础床号**：末 2 位整数，如 `01`。

**识别流程（可靠流程）**：
1. **设定解析模板**：用户用模板定义床号分段。默认模板正则：`^(\d{3})([A-Z])([A-Z]{0,2})?(\d{2})$`
   - 可配置项：各段长度、哪一段组合为「病区」、特殊床标记字符集合（默认 `J`、`YZ`）。
2. **导入即解析**：批量导入（4.1.3）时按模板逐条解析，产出每床的 `病区 / 特殊类型 / 基础床号`。
3. **自动分类**：特殊段为空 → 判定为**基础真实床**；含特殊标记 → 判定为**加床/特殊床（J/YZ）**，其「真实/虚拟」待 Step1.4 界定。
4. **预览与人工校正（界定真实/虚拟）**：解析后展示预览表（床号 | 病区 | 类型 | 床号 | 判定），用户逐条界定每个加床是「真实（查房走访）」还是「虚拟（不走访）」。内置示例中 `J04/J06/J07/J08/J09/J10/J12/J13/J14/J15/J16/J17/J18` 为真实加床；`J01/J02/J03/J05/J11…` 等未出现的床号为虚拟床。确认后生效。
5. **真实加床的房间归属**：真实加床（如 `J04`）物理上位于某病房旁，需指定其归属病房——内置示例由序列位置体现（`J04` 列于 41-43 房之后，即归属该房）；自定义时由用户在识别设置中**手动指定**。虚拟床无物理位置，不纳入查房序列，亦无需房间归属。

#### 4.9.3 查房顺序设置（按病房展开，Step 2：列表展示）

**术语澄清**：
- **基础床号（base bed number）**：床号经 4.9.2 解析后的**末位整数**，格式化为 2 位，如 `01`、`23`。它是查房序列的最小排序单位，与病区前缀（如 `309W`）无关。
- **普通病床（regular bed）**：指 `bedType === 'real'` 的**真实床**，不含真实加床（extra-real）与虚拟床（virtual）。下述「病床数量」均指普通病床。

**两类内置配置（应用初始并不知道真实用户的床号类型，仅提供通用规则）**：
- **① 默认规则（default rule）**：即前文 4.9.2 的完整示例——以 `309W01` 系列为代表的、带病区前缀的具体病房布局（项目已实现，见 `defaultRoundingOrder()`）。它是开箱即用的「示范」。
- **② 基础规则（basic rule）**：**只有基础床号**，不含病区前缀，如 `01`、`02`、`03`……。一种与具体病区解耦的通用布局，适用于尚未导入真实床号、或希望按纯编号排班的场景。

**设置页规则态（rule type）**：查房顺序设置页需同时存在三种态，并以标识呈现当前所用规则：
- **默认规则 / 基础规则 / 自定义规则**。
- **切换为自定义的判定**：当用户对「默认规则」或「基础规则」中的任意内容（病房块、床号、加床、拖拽重排、导入覆盖等）进行**修改**，其规则标识**自动从「默认/基础」变为「自定义」**——任何手动改动都视为自定义，不再归属于内置规则。

**基础规则设置流程（从零开始）**：
1. **确认普通病床数量**：开始设置前，先与用户确认「普通病床数量」（仅 real 床，不含加床/虚拟床）。例如 `40`。
2. **确认平均单一病房床数**：再确认每间病房的平均床位数，如 `3`。
3. **推导病房块数量**：`病房数 = ceil(普通病床数量 / 平均单一病房床数)`，据此建立初始病房块（最后一间房床数可能少于平均值）。
4. **填充病房块**：每个病房块由连续的**基础床号**构成，如第 1 房 = `{01, 02, 03}`、第 2 房 = `{04, 05, 06}`……依次递增，总数恰为普通病床数。
5. **病房块内可编辑**：病房块内的**基础床号集合**与**病床数量**均可修改——可单独增加某床号、删除某床号（如某房实际只有 2 床，去掉 `03`）；修改后块内床号按升序重排。

**真实加床块（extra bed block）**：
- 内容与「病房块」类似（也是一组床号组成的块），但**由用户自行添加**，系统不自动生成。
- 块内放真实加床床号（如 `J04`、`J06`）；可在序列任意位置插入，独立于普通病房块。
- 虚拟床不纳入任何块。

**默认规则的展示与调整**：
- 选择「默认规则」时，须以**基础规则的同款块样式**进行**正确、完整的展示**（同样以「病房块 + 真实加床块」呈现，病房块内为 `309W01` 等带前缀床号），让用户看到完整布局并可调整。
- 一旦调整，规则标识自动转为「自定义」（见上）。

**完整导入 / 导出（复制文本格式）**：
- 查房顺序支持**完整导入与导出**，以「可复制文本」形式（多行文本或 JSON），便于备份、跨设备粘贴、或分享给同事。
- 导出文本可在导入时**完整还原**为当前规则（含规则态、病房块、加床块、床号）。
- 导入时做基础校验（床号去空、基础床号升序规整），非法内容提示后忽略。

**排序规则**：
- 被设置的床号（基础床号 / 完整床号）**一律按从小到大（升序）排列**；病房块内、块间、导出文本均遵循升序。

**正序 / 反序切换（列表方向）**：
- 设置好的查房顺序在**首页病人列表**上支持**一键切换「正序 / 反序」**。正序 = 用户编排的顺序（从前往后）；反序 = 将整条展示序列**整体反转**（含组合卡片顺序与卡片内床号顺序），用于从尾到头逆向查房。
- 该切换**属于首页病人列表的展示偏好，与「查房顺序设置」完全解耦**：它不存放在 `RoundingConfig`，不出现在查房顺序设置页，不修改任何块结构或床号编排；仅在首页对 `resolveOrder` 输出的最终列表做整体反转后展示。
- 以持久化字段 `listDirection: 'forward' | 'reverse'` 存储于 `Settings`（默认 `'forward'`）。首页提供「正序 / 反序」分段控件，切换即生效并持久化；随时切回正序即恢复原始编排，查房顺序设置内容不受影响。

**与病人列表的关系**：病人列表（4.1.1）排序以本序列的 `roundingOrder` 为准（并受首页 `listDirection` 影响）；未进入序列的病人按「病区 + 基础床号数值」升序追加到末尾。

##### 4.9.3.1 设计动机与交互原则（v2.6 修订）
原方案以「病区 Tab + 范围输入」组织查房顺序，仍隐含「先有病区、再填范围」的前提，且未区分默认/基础/自定义三态，与用户实际「不知道真实床号类型、只按基础编号排班」的工作方式不符。改进后：
1. **基础床号是第一公民**：序列的最小单位是解析后的基础床号（`01`/`23`），病区前缀只是其展示形式之一，而非必要条件。
2. **两内置规则 + 自定义三态**：默认规则（带病区示范）与基础规则（纯基础床号）开箱即用；任何修改自动归为自定义，状态清晰可追溯。
3. **从数量推导布局**：基础规则由「普通病床数 ÷ 平均病房床数」自动生成病房块，零手输床号；病房块内床号可增删微调。
4. **块即单元**：病房块与真实加床块是统一的「块」概念，真实加床由用户自行成块，灵活穿插。
5. **可复制的完整导入/导出**：整份规则可一键导出为文本、再导入还原，便于流转与备份。
6. **升序为纲**：所有床号升序排列，保证顺序稳定、可读。

#### 4.9.4 数据模型字段

床号识别结果需持久化到数据模型，供排序与展示使用（完整 Schema 见第 7 章）：

- **Patient** 字段（见 7.1）：
  ```typescript
  ward: string      // 病区，如 "309W"（由 4.9.2 识别得出；基础规则下可空）
  room: string      // 病房标识，如 "309W41-43" 或 "Room 3"（同房内床共享，由查房序列块归属得出）
  bedBase: number   // 基础床号数值，如 23（由 4.9.2 解析，即 4.9.3 所述「基础床号」）
  bedType: 'real' | 'extra-real' | 'virtual'  // 真实床(普通病床) / 真实加床 / 虚拟床
  ```
- **Settings.roundingOrder**（v2.6 重构为块模型 `RoundingConfig`，取代原 `Unit[]`）：
  ```typescript
  interface RoundingConfig {
    ruleType: 'default' | 'basic' | 'custom';  // 当前规则态；修改内置规则后自动置 'custom'
    regularBedCount?: number;   // 基础规则：普通病床总数（不含加床/虚拟）
    avgBedsPerRoom?: number;    // 基础规则：平均单一病房床数
    blocks: RoundingBlock[];    // 有序序列：病房块 + 真实加床块（顺序本身即查房顺序）
  }
  // 列表方向（正/反序）属于首页展示偏好，独立存放于 Settings（见 7.1）：
  //   listDirection?: 'forward' | 'reverse'  // 默认 'forward'，仅首页病人列表使用
  type RoundingBlock =
    | { id: string; kind: 'room';  ward?: string; beds: string[] }  // 病房块：beds 为基础床号(如["01","02","03"]) 或 带前缀床号(如["309W01",...])
    | { id: string; kind: 'extra';            beds: string[] };    // 真实加床块：beds 为真实加床床号(如["309WJ04"]或["J04"])
  ```
  - `beds` 内元素**升序**存储；`ruleType === 'basic'` 时块不含 `ward`，床号仅为基础床号。
  - 兼容迁移：旧 `Unit[]` 数据在首次读取时按 `kind` 映射为 `RoundingBlock[]`，并补 `id` 与 `ruleType`（默认 `'custom'`）。

### 5.10 病人卡片

- 卡片采用 `bg-card` 背景、`rounded-xl` 圆角、`border border-border/50` 边框，无 hover 时无阴影。
- 布局为横向 Flex：左侧床号色块 → 中部信息 → 右侧操作菜单入口。
- 左侧床号色块：
  - 尺寸 `w-10 h-10`，圆角 `rounded-lg`。
  - 背景色为 `patient.groupColor`，文字色通过 `contrastTextColor` 计算。
  - 显示床号末两位数字；若床号不含数字，则显示前 3 位。
- 中部信息：
  - 姓名使用 `font-semibold text-main text-[14px]`。
  - 诊断使用 `text-muted text-[12px]` 并截断。
  - 分组标签：背景色为分组颜色加 20% 透明度，文字色为分组颜色。

- 操作入口：
  - 桌面端 hover 时显示右侧 `MoreHorizontal` 按钮；移动端通过长按 500ms 触发菜单。
  - 菜单以居中 Dialog 形式展现，选项：更改分组、删除病人。

- 点击卡片进入 `/patient/{id}`；点击菜单按钮时长按状态会阻止跳转。

### 5.11 底部面板

- 用于承载添加/编辑表单、批量导入、通用待办等输入型任务。

- 顶部显示 8px 宽、圆角抓手条（`bg-muted/15`），提示可拖拽关闭。
- 背景遮罩为 `bg-black/30 backdrop-blur-sm`，点击遮罩或关闭按钮关闭。
- 打开动画：`initial={{ y: '100%' }}`，`animate={{ y: 0 }}`，使用 spring 动画（damping 28, stiffness 320）。

- 内容区默认 `p-5`，可滚动，超出最大高度时显示滚动条（使用 `scrollbar-hide` 隐藏原生滚动条）。
- 标题栏左侧显示标题，右侧显示 `X` 关闭按钮。
## 6. 视觉设计系统

### 6.1 设计原则

1. **温暖专业** - 使用暖色调（陶土色、米白色）营造医疗场景的信任感

### 6.11 CSS 工具类与自定义组件

CSS 工具类定义于 `app/globals.css`，按 `@layer` 组织。

**@layer base**
- 为 `:root` 和 `.dark` 分别定义语义化 RGB 变量。
- `body` 使用 `bg-surface text-main antialiased`。
- 启用 `font-feature-settings: "cv02", "cv03", "cv04", "cv11"` 优化数字和标点。
- 统一 `input, textarea, select` 字体大小为 16px，避免 iOS 自动放大页面。
- 移除按钮点击时的高亮。

**@layer components**
- `.input`：全宽、圆角 12px、卡片背景、边框、内边距 4/3、聚焦时主色边框 + 2px 主色光晕。
- `.btn-primary`：主色填充、白色文字、12px 圆角、小阴影、hover 阴影加深、active 缩放 0.97、禁用 50% 透明度。
- `.btn-secondary`：卡片背景、边框、主色文字、12px 圆角、hover 边框加深。
- `.card`：卡片背景、12px 圆角、浅色边框、`.shadow-xs`。
- `.card-interactive`：在 `.card` 基础上增加 hover 阴影加深与边框加深，用于可点击卡片。
- `.badge`：行内 flex、6px 圆角、10px 加粗文字。
- `.badge-primary/success/warning/danger/muted`：语义色**深色调**文字 + 浅色实底（surface），确保文字/背景对比度 ≥ 4.5:1（warning 浅色用 `#8a5a12`、深色用 `#e0a85a`）。

**@layer utilities**
- `.bg-surface/.text-main/.bg-card/.text-muted/.border-custom/.text-primary/.bg-primary/.bg-primary-hover`：映射到 CSS 变量。
- `.shadow-xs/sm/md/lg/xl`：精确阴影值。
- `.shadow-primary-sm/md`：主色光晕阴影。
- `.safe-area-pb`：底部安全区内边距 `env(safe-area-inset-bottom, 0px)`。
- `.scrollbar-hide`：隐藏滚动条（含 IE、Firefox、WebKit）。

**自定义动画**
- `fadeIn`：0.2s ease-out，透明度 0 → 1。
- `slideUp`：0.3s cubic-bezier(0.16, 1, 0.3, 1)，向下偏移 12px → 0。
- `slideDown`：0.3s cubic-bezier(0.16, 1, 0.3, 1)，向上偏移 12px → 0。
- `scaleIn`：0.2s cubic-bezier(0.16, 1, 0.3, 1)，缩放 0.92 → 1。
- Tailwind 配置中同步定义 `animate-fade-in`、`animate-slide-up`、`animate-scale-in`，便于 Tailwind 类名使用。

## 9. 动画与动效

### 9.1 动效原则

1. **有意义** - 动画用于引导注意力或反馈操作结果
2. **克制** - 避免过多装饰性动画
3. **快速** - 大多数动画 200-300ms
4. **流畅** - 60fps，使用 transform 和 opacity
5. **可禁用** - 尊重 `prefers-reduced-motion` 设置

### 9.2 页面过渡

- Tab 切换：无过渡，直接切换
- 详情页进入：从底部滑入（300ms，ease-out）
- 详情页返回：向下滑出（200ms，ease-in）
- 设置页进入：从右侧滑入
- 实现说明：上述 route 级转场需基于 View Transitions API 或 Framer Motion 的 `AnimatePresence` 封装页面容器实现（Next.js App Router + Static Export 原生不支持页面级动画）。

### 9.3 组件动效

**列表项**：
- 进入：stagger slide-up，间隔 30ms
- 删除：向左滑出 + 高度收缩
- 新增：scale-in

**Bottom Sheet**：
- 打开：遮罩 fade-in，sheet slide-up（300ms）
- 关闭：sheet slide-down，遮罩 fade-out（200ms）

**Header 操作按钮**：
- 点击：缩放 0.97（轻量反馈），无展开菜单动画

**Toast**：
- 进入：从顶部/底部 slide-in（200ms）
- 退出：fade-out（150ms）
- 自动消失：3 秒

**Dialog**：
- 打开：scale-in + fade-in
- 关闭：fade-out

### 9.4 手势反馈

- 点击按钮：缩放 0.97
- 长按卡片：轻微放大 + 震动
- 左滑待办：阻力感，超过阈值后触发
- 下拉刷新：弹性回弹

### 9.5 性能要求

- 所有动画使用 transform/opacity
- 避免布局属性动画
- 列表滚动使用虚拟化（如病人数 > 100）
- 动画帧率 >= 60fps

## 10. 通知与提醒

### 10.1 提醒触发规则

**启动检查**：
- 每次打开 App 时检查所有待办
- 找出今天到期和已逾期的待办

**定时检查**：
- 每 1 小时检查一次（后台使用 setInterval）
- 仅当页面活跃时运行

**换药/查血提醒**：
- 根据 `lastDressingChange` 和 `dressingFrequency` 计算下次换药日期
- 根据 `bloodTestDay` 计算下次查血日期
- 如果日期为今天，加入提醒

### 10.2 提醒栏

**位置**：首页顶部
**显示条件**：
- 存在今天到期或已逾期的待办
- 存在需要换药/查血的病人

**样式**：
- danger：红色背景，"X 个待办已逾期 / Y 人需换药"
- warning：橙色背景，"今天有 X 个待办到期 / Y 人需查血"
- 点击跳转对应筛选列表

### 10.3 浏览器通知

**触发条件**：
- 用户授权通知权限
- 定时检查时发现新的到期待办
- 同一待办不重复通知

**通知内容**：
- 标题："临床助手"
- 正文："今天有 X 个待办到期"

### 10.4 静默规则

- 用户未授权时不弹出系统提示
- 已完成待办不提醒
- 通知发送后 1 小时内不重复
- 应用在前台时优先使用应用内提醒栏，不发送系统通知

- Service Worker 注册失败时，应用仍可正常使用

### 11.3 离线策略

- 数据存储在 IndexedDB
- 导入/导出使用本地文件 API
- 离线时不显示需要网络的功能

### 11.4 存储管理

- 使用 Dexie.js 封装 IndexedDB
- 数据库名：`ClinicalDB`

## 14. 边界情况与异常处理

### 14.1 数据边界

- 病人数上限：10,000（设计上限，未压测）
- 单个病人待办上限：1,000（设计上限，未压测）
- 待办内容长度上限：200 字符
- 分组名上限：20 字符
- 超过上限时提示用户清理数据

### 14.2 交互边界

- 删除病人时：如果存在未完成待办，二次确认提示"该病人还有 X 个未完成待办"；确认删除后弹 Toast「已删除 · 撤销」，点击可恢复病人及其全部待办（约 5 秒窗口）
- 删除待办：立即从列表移除，同时弹 Toast「已删除 · 撤销」，点击撤销可恢复（约 5 秒窗口）
- 重复导入：按姓名更新，不重复创建
- 离线操作：正常保存，网络恢复后无需同步

### 14.3 解析边界

- 时间解析失败：允许保存，标记为"无日期"
- 多个时间：取优先级最高的第一个
- 过期日期：仍允许保存，显示"已逾期"
- 无法识别的日期：如"大后天"，显示"未识别"

### 14.4 导入边界

- 空文本导入：提示"请输入导入内容"
- 格式错误行（缺床号/姓名等）：跳过该行，并在预览中列出被跳过行号与原因，用户可修正后重新预览
- 重复姓名：更新信息，不报错
