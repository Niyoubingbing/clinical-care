## 1. 产品概述

### 1.1 产品定位

**临床病人管理助手** 是一款面向临床医生的移动端 PWA 应用，用于管理住院病人的查房信息、待办事项和医嘱记录。

**核心价值**：
- 📱 **移动优先** - 手机上使用，随时随地记录
- ⚡ **秒开** - 本地存储，无网络也能用
- 🤖 **智能** - 自动解析待办时间，智能提醒
- 📋 **高效** - 批量导入病人，快捷操作

### 1.2 产品边界

**包含**：
- ✅ 病人信息管理（增删改查）
- ✅ 待办事项管理（智能时间解析 + 提醒）
- ✅ 查房列表（自定义排序）
- ✅ 批量导入/导出
- ✅ 快捷操作
- ✅ 每日小结

**不包含**（进阶功能，未来考虑）：
- ❌ 多用户协作
- ❌ 统计分析/报表
- ❌ 云端同步
- ❌ 原生 App

### 1.3 设计原则

1. **效率优先** - 每次操作不超过 3 步，查房场景下可单手完成
2. **容错设计** - 所有删除操作可撤销，导入前可预览
3. **离线可用** - 不依赖网络，任何场景都能使用
4. **信息密度** - 一页显示 40+ 病人摘要，关键信息一目了然

### 1.4 核心术语

| 术语 | 解释 |
|------|------|
| 查房 | 医生按固定顺序巡视病人 |
| 待办 | 需要完成的任务，含时间解析后的目标日期 |
| 提醒 | 基于目标日期生成的到期/逾期提示，以及根据换药频率/查血日计算的需换药、需查血提示 |
| 分组 | 病区的医疗组（如解组、勇组），带颜色标签 |
| 快捷待办 | 设置中自定义的常用待办模板 |
| 每日小结 | 当天完成的待办和换药/查血记录汇总 |

## 2. 用户画像

### 2.1 目标用户

| 属性 | 描述 |
|------|------|
| **角色** | 临床医生（住院医师/主治医师） |
| **科室** | 骨科（可扩展到其他外科科室） |
| **技术水平** | 熟练使用智能手机 |
| **使用场景** | 查房时、办公室、家中 |

### 2.2 用户痛点

1. **病人多，容易漏** - 40+ 病人，每天哪些要查血、哪些要换药，容易忘记
2. **待办分散** - 每个病人的待办不同，难以统一管理
3. **时间感知弱** - "星期二出院"这种待办，到了当天容易忘记
4. **数据录入慢** - 每天要重新输入病人信息，效率低

### 2.3 用户目标

1. **快速记录** - 查房时快速记录每个病人的待办
2. **智能提醒** - 系统自动提醒今天要做的事
3. **高效导入** - 粘贴最短文本，快速同步全量数据
4. **每日小结** - 快速生成当天完成的事项，用于交班/记录

### 2.4 使用场景矩阵

| 场景 | 时间 | 设备 | 目标 | 关键功能 |
|------|------|------|------|----------|
| 早晨查房 | 08:00-10:00 | 手机 | 查看今日提醒并按顺序查房 | 提醒栏、查房列表 |
| 查房记录 | 10:00-11:00 | 手机 | 快速添加/完成待办 | 快捷待办、病人详情 |
| 办公室整理 | 14:00-15:00 | 桌面/平板 | 整理待办、调整顺序 | 待办页、设置 |
| 交接班 | 17:00-18:00 | 任意 | 生成今日小结 | 每日小结 |
| 每日更新名单 | 查房前（每日） | 手机/桌面 | 更新当天病人列表（粘贴最新 roster，按姓名匹配增改） | 批量导入 |

## 3. 核心工作流

### 3.1 每日工作流

```
早晨查房前
  ↓
更新当天病人列表
（粘贴最新 roster，按姓名匹配增改）
  ↓
打开 App，查看今日提醒
（查血、换药、待办到期）
  ↓
查房（按自定义顺序）
  ↓
给每个病人添加/更新待办
（系统自动解析时间，到期提醒）
  ↓
查房结束，生成每日小结
（复制给上级/交班）
  ↓
随时查看待办，完成打勾
```

### 3.2 备份工作流

```
每日查房前（详见 3.1）
  ↓
更新当天病人列表（粘贴 roster，按姓名匹配增改）
  ↓
每天查房，添加/更新待办
  ↓
（按需，建议每周）导出数据备份
（防止丢失）
```

### 3.3 病人全生命周期

```
更新名单（批量导入）/ 手动添加
  ↓
维护基本信息（床号、诊断、分组、手术日期）
  ↓
查房过程中添加/完成待办
  ↓
系统根据换药频率/查血日自动提醒
  ↓
病人出院或转科 → 手动删除
```

### 3.4 待办全生命周期

```
输入待办文本
  ↓
实时时间解析（今明后天/星期几/具体日期）
  ↓
保存到病人名下
  ↓
首页/待办页显示剩余天数
  ↓
到期当天高亮提醒
  ↓
完成后打勾
  ↓
每日小结统计已完成项
```

## 4. 功能需求

### 4.1 病人管理

#### 4.1.1 病人列表
**功能描述**：
- 显示所有病人，按**自定义查房顺序**（`Settings.roundingOrder`）排序。
- 支持**分组筛选**（如：解组、勇组）。
- 支持**搜索**（床号、姓名、诊断）。
- 每个病人卡片显示：
  - 左侧床号色块（取床号末两位数字或前 3 位，背景为分组颜色）。
  - 中部姓名、诊断。
  - 分组名称标签（带颜色）。
  - 未完成待办数量。
  - 逾期/今日到期/需换药/需查血状态徽章。

**排序规则**：
1. 按 `roundingOrder`（查房序列，见 4.9.3）的单元顺序排序；病房块内按基础床号升序。
2. 未进入查房序列的病人，按「病区 + 基础床号数值」升序追加到末尾（病区与基础床号由 4.9.2 识别得出，如 `309W23` → 病区 309W、床号 23）。
3. 启用分组筛选时，列表只显示该分组病人，仍按上述规则排序。
4. 同一病人不重复显示。

**交互细节**：
- 点击卡片 → 进入 `/patient/{id}` 详情页。
- 长按卡片（移动端）或点击右侧 `...` → 显示操作菜单（更改分组、删除）。
- 搜索支持**拼音首字母**（依赖 `pinyin-pro`），同时匹配原始文本。
- 搜索输入为空时显示全部病人，无额外历史记录。
- 空搜索结果时显示“暂无病人”提示。

**字段说明**（病人卡片中展示/参与提醒）：
- 床号：必填，用于排序和识别。
- 姓名：必填，用于批量导入匹配。
- 诊断：必填。
- 分组：可选，带颜色标签。
- 手术日期：可选，YYYY-MM-DD。
- 换药频率：可选，1-90 整数天。
- 上次换药日期：可选，YYYY-MM-DD。
- 查血日期：可选，可填写多个周几名称（如“周一 周三”）。

**验收标准**：
- [ ] 列表加载时间 < 500ms（40 个病人）。
- [ ] 搜索响应时间 < 100ms。
- [ ] 拼音搜索准确率 > 95%。

#### 4.1.2 添加/编辑病人
**功能描述**：
- 在 Bottom Sheet 中手动添加或编辑病人信息。
- 字段：床号、姓名、诊断、分组名称、分组颜色、手术日期、换药频率（天）、上次换药日期、查血日期。

**交互细节**：
- 分组输入提供 `解组、勇组、李组、王组` 预设选项。
- 分组颜色使用原生 `<input type="color">`，默认取预设颜色列表。
- 保存时校验床号、姓名、诊断是否为空，否则提示。
- 新增病人时**不**自动加入 `roundingOrder`，未在顺序中的病人按床号自然顺序排在后面。
- 编辑床号后**不**同步修改 `roundingOrder`，需要用户手动调整。
- 编辑状态下提供“删除病人”按钮，删除后同步删除该病人的所有待办。

**表单校验**：
- 床号、姓名、诊断为必填。
- 换药频率（若填写）应为整数。
- 日期格式使用原生日期选择器，保证格式正确。

**验收标准**：
- [ ] 保存后立即更新列表。
- [ ] 分组颜色实时更新。
- [ ] 支持删除病人并级联删除待办。

#### 4.1.3 批量导入病人
**功能描述**：
- 在首页 Header「添加病人」中进入“批量导入”（Sheet 内「批量导入」tab），粘贴文本后预览并导入。
- 文本格式：`床号 姓名 诊断`（如 `309W23 魏亿鑫 取除骨折内固定装置`）。
- 导入逻辑：
  - 按**姓名**精确匹配现有病人，匹配到则更新床号、诊断。
  - 未匹配到则新增，默认分组为空、分组颜色 `#e2e8f0`。
  - 导入文本中未出现的病人，默认**保留**；仅在导入预览中显式勾选“移除未出现病人”时才归入“删除”，确认后删除并级联删除其待办。预览中展示将被删除病人的未完成待办数量。

**交互细节**：
- 导入前显示预览：新增 / 更新 / 删除的人数及明细列表。
- 若用户勾选“移除未出现病人”且存在待删除病人，弹出二次确认对话框，并显示各病人的未完成待办数量。
- 导入后 Toast 提示：新增 X 人、更新 X 人、删除 X 人。
- 文本框始终保留，支持修改后重新预览。

**文本解析规则**：
- 每行一个病人。
- 字段以空白字符（空格或制表符）分隔，至少 3 列：床号、姓名、诊断。
- 第三列及以后合并为诊断。
- 忽略空行与前后空白。
- 行中缺少姓名或床号时跳过，并在预览中列出被跳过行号与原因。
- 姓名匹配大小写敏感。

**验收标准**：
- [ ] 导入 40 个病人 < 3 秒。
- [ ] 同名病人正确更新，不重复创建。
- [ ] 删除确认弹窗清晰明了。

### 4.2 待办管理

#### 4.2.1 添加待办
**功能描述**：
- 每个病人可以有多个待办。
- 待办内容：自由文本（如“星期二出院”、“后天换药”、“周四复查X光”）。
- 待办类型：换药、查血、开术前、明天出院、康复会诊、会诊、复查、开查血、其他。
- 系统基于 `lib/time-parser` 自动解析文本中的时间。

**智能时间解析**：
- 支持识别：
  - 今明后天：今天、明天、后天（注意：暂不支持“大后天”）。
  - 星期：周日~周六、星期日到星期六、周X、下星期X、下礼拜X、下下X。
  - 具体日期：YYYY-MM-DD、YYYY/MM/DD、MM/DD、MM-DD、M月D日。
- 解析优先级：今明后天 > 星期 > 具体日期。
- 解析成功后计算 `dueDate` 和剩余天数。

**实现细节**：
- 星期解析规则：以当前日期为基准，计算目标星期与今日星期差值；若文本包含“下星期/下礼拜/下下”且差值为 0，则返回 7 天后（下周一）而非今天。
- 当文本同时包含多种时间表达时，取优先级最高的第一个结果（如“明天星期二”会识别为“明天”）。
- 具体日期省略年份时，默认使用当前年份。
- 解析成功返回 `{ date: 'YYYY-MM-DD', label: '今明后天' | '星期几' | '具体日期' }`，失败返回 `null`。
- 实时解析通过 `useEffect` 监听输入变化，延迟由 React 渲染周期决定，通常 < 100ms。

**交互细节**：
- 添加待办使用 Bottom Sheet 中的表单。
- 内容输入框为 textarea，输入时实时调用 `parseTime` 显示解析结果。
- 解析成功时显示：`识别到时间：{date}（{label}）`。
- 解析失败时显示：`未识别到时间`。
- 提供类型下拉选择框（默认值“其他”）。
- 快捷待办按钮点击时，自动以该快捷待办的内容和类型创建待办。

**验收标准**：
- [ ] 时间解析准确率 > 90%。
- [ ] 解析结果实时显示（< 300ms）。
- [ ] 支持三种格式。

#### 4.2.2 待办列表
**功能描述**：
- 待办页显示所有病人的待办，支持分段筛选：
  - 全部 / 未完成 / 已完成 / 今天到期 / 已逾期。
- 待办页顶部新增分段切换：**病人待办 / 通用待办**。
  - 「病人待办」：现有全部病人待办，沿用上述筛选。
  - 「通用待办」：仅显示 `patientId` 为空的通用待办（见 4.8）。
- 点击病人待办进入对应病人详情页；通用待办点击可查看/完成/删除，不跳转病人页。

**排序规则**：
1. 未完成优先，逾期/今日到期待办权重更高。
2. 按 `dueDate` 升序（先到期的在前）。
3. 无日期的待办排在最后。
4. 同一天内按 `createdAt` 倒序（最新创建的在前）。

**交互细节**：
- 每个待办显示：
  - 病人床号与姓名（待办页）。
  - 待办内容。
  - 剩余天数标签（还剩 X 天 / 今天到期 / 已逾期 X 天）。
  - 状态复选框。
  - 类型徽章（换药、查血等）。
- 左滑待办项露出右侧“完成”与“删除”按钮。
  - 左滑水平位移超过 50px 时展开操作区，露出宽度 7rem。
  - 右滑超过 50px 或点击内容区域收起。
  - 操作按钮为圆形按钮：完成（绿色 success）、删除（红色 danger）。
- 点击复选框直接切换完成状态。
- 已完成待办显示删除线与 60% 透明度，数据保留。

**验收标准**：
- [ ] 待办列表加载 < 500ms。
- [ ] 到期待办高亮显示。
- [ ] 完成待办后列表实时更新。

#### 4.2.3 智能提醒
**功能描述**：
- 每次打开应用时检查待办、换药、查血状态。
- 在首页顶部显示提醒栏，汇总今日到期、已逾期、需换药、需查血的数量。

**提醒优先级**：

### 4.9 查房顺序与床号管理

> 本模块集中定义「床号如何解析」与「查房序列如何生成」两大问题。床号是医院层级里的最小单位，真实编码常复合了「病区—病房—病床（含各类型）」多层信息；而查房是按「病房（房间）」逐房展开的。本模块提供一套**可配置、通用**的识别方案，以及内置示例排序，使系统既能开箱即用，也能适配不同医院的床号编码。
> 配置入口位于设置页的「床号识别」与「查房顺序」项（见 4.6.2、4.6.3）；本模块为完整功能定义。

#### 4.9.1 功能概述

- **为什么需要**：基础床号本应是 `1/2/3…/40` 这样的整数顺序，但医院里因不同病区、不同类型病床（普通床、抢救床、隔离床、加床等），出现了复杂编码来定位「这张床在医院的哪个位置」。系统必须能从中识别出「病区 / 病房 / 病床」三层，才能正确排序与展示。
- **两个核心步骤**：
  - **Step 1（识别）**：从导入的床号文本中解析出「病区、病房、病床」，并界定每张床是真实床 / 真实加床 / 虚拟床（见 4.9.2）。
  - **Step 2（展示）**：针对真实床，按「病房在前、房内床在后」组织查房序列，并提供内置示例与自定义拖拽（见 4.9.3）。
- **与每日工作流的关系**：每日查房前更新病人名单（见 3.1），导入即触发 Step 1 识别；查房序列（Step 2）决定病人列表（4.1.1）的展示顺序。

#### 4.9.2 床号结构模型与通用识别（Step 1：找到所有真实床）

**床号结构模型**（以默认示例 `309W01` / `309WJ01` / `309E01` / `309WYZ01` 为例）：
```
[病区基底 3位数字][病区方位 1位字母][特殊床标记 0~2位字母][基础床号 2位数字]
    309              W              (空=真实床 / J / YZ)        01
```
- **病区（Ward）** = 病区基底 + 病区方位，如 `309W`、`309E`。`309` 代表楼栋/楼层，`W/E` 代表西/东病区。
- **特殊床标记**：可选。`J` = 病房额外床（加床），由「真实床 + 虚拟床」组成——真实床有物理位置（查房时走访），虚拟床无物理位置（不走访）；`YZ` = 另一种形式的虚拟床（无物理位置，不走访）。无标记者为基础**真实床**（房内床）。
- **基础床号**：末 2 位整数，如 `01`。

**识别流程（可靠流程）**：
1. **设定解析模板**：用户用模板定义床号分段。默认模板正则：`^(\d{3})([A-Z])([A-Z]{0,2})?(\d{2})$`
   - 可配置项：各段长度、哪一段组合为「病区」、特殊床标记字符集合（默认 `J`、`YZ`）。
2. **导入即解析**：批量导入（4.1.3）时按模板逐条解析，产出每床的 `病区 / 特殊类型 / 基础床号`。
3. **自动分类**：特殊段为空 → 判定为**基础真实床**；含特殊标记 → 判定为**加床/特殊床（J/YZ）**，其「真实/虚拟」待 Step1.4 界定。
4. **预览与人工校正（界定真实/虚拟）**：解析后展示预览表（床号 | 病区 | 类型 | 床号 | 判定），用户逐条界定每个加床是「真实（查房走访）」还是「虚拟（不走访）」。内置示例中 `J04/J06/J07/J08/J09/J10/J12/J13/J14/J15/J16/J17/J18` 为真实加床；`J01/J02/J03/J05/J11…` 等未出现的床号为虚拟床。确认后生效。
5. **真实加床的房间归属**：真实加床（如 `J04`）物理上位于某病房旁，需指定其归属病房——内置示例由序列位置体现（`J04` 列于 41-43 房之后，即归属该房）；自定义时由用户在识别设置中**手动指定**。虚拟床无物理位置，不纳入查房序列，亦无需房间归属。

#### 4.9.3 查房顺序设置（按病房展开，Step 2：列表展示）

**设计目标**：查房序列按「病房（房间）」展开——病区由一个个房间组成，每间房 2–4 张真实床，查房逐房进行；特殊床（J/YZ）穿插其间。设置页需提供（1）清晰的内置默认顺序、（2）比纯文本框更易操作的自定义方式。

**数据模型**：查房序列 = 一组有序「单元（Unit）」。每个单元为：
- **病房（Room）**：含 1–4 张真实床（房内基础床，及/或真实加床），归属某病区；
- **真实加床（Extra real bed）**：单张 J/YZ 类型的真实床（如 `J04`），物理位于某病房旁，作为独立可拖单元穿插在序列中；
- （虚拟床无物理位置，不纳入查房序列。）

**内置示例排序（默认，病区 309W）**（病房块在前，方括号 `[Jxx]` 表示穿插其前的**真实加床**（非虚拟床），物理上位于该病房旁）：
```
(309W41, 309W42, 309W43)   [J04]
(309W38, 309W39, 309W40)   [J06] [J07]
(309W36, 309W37)
(309W01, 309W02, 309W03)
(309W04, 309W05, 309W06)   [J08]
(309W34, 309W35)
(309W07, 309W08, 309W09)   [J09] [J10] [J12]
(309W10, 309W11, 309W12)   [J13]
(309W13, 309W14, 309W15)   [J14]
(309W16, 309W17, 309W18)   [J15]
(309W19, 309W20, 309W21)   [J16]
(309W22, 309W23, 309W24)   [J17]
(309W25, 309W26, 309W27)   [J18]
(309W28, 309W29, 309W30)
(309W31, 309W32, 309W33)
```

**自定义顺序（设置页交互）**：
- 设置页提供独立、交互更佳的「查房顺序」编辑页，以「病房块 + 真实加床」为单位进行**结构化编辑**与拖拽重排。
- **病房块：以「病区 + 基础床号范围」结构化生成（替代纯文本输入）**：
  - 病区：从已识别病区（病人库 / 识别设置）下拉选择，或手动输入基底（如 `309W`）。
  - 床号：以「起始—结束」数字（`01`–`40`）或一组可点选的床号 chips（`01 02 … 40`）指定该病房真实床；系统按 4.9.2 模板规则自动展开为完整床号（如 `309W01 … 309W40`）并写入单元。
  - 编辑区实时预览展开后的床号序列，所见即所得。
- **真实加床：从已识别清单选择（替代手输床号 + 归属）**：
  - 列出已识别为「真实加床（extra-real）」的床号（来源于病人库 / 4.9.2 识别预览），用户勾选并下拉指定归属病房；虚拟床不可选、不纳入序列。
  - 上述病区、床号、类型均复用 4.9.2 已解析的 `ward / bedBase / bedType`，无需重复录入。
- **从病人库生成草稿（v2.3 新增）**：提供「按当前病人库生成」入口，读取全部病人，按「病区 → 基础床号」自动分组排序，生成病房块序列与真实加床归属草稿，作为自定义起点；生成后用户仍可微调、拖拽。呼应「床号已解析，应可直接用于设置顺序」的诉求，并仍遵循 PRD「新增病人不自动加入 `roundingOrder`」原则（此为手动触发，非自动同步）。
- **拖拽重排**：病房块整体拖拽（移动一房 = 移动其中所有真实床，房内床相对顺序保持，可单独微调）；真实加床独立拖拽插入任意位置；移动端长按、桌面端拖拽 + 键盘可达。
- **高级文本批量输入（保留为折叠选项）**：每行一个床号，保存时按行生成单元（相邻真实床自动成组为一病房块），去空行、不强制去重。作为结构化编辑的补充，而非主交互。
- 提供「重置为内置示例」按钮恢复默认；每日导入新增、序列中尚无的床默认追加到末尾，用户再拖入合适位置。
- 除内置示例外，用户可在「识别设置」基础上为新病区自定义病房分组与顺序（通用方案，不限于 309W）。

##### 4.9.3.1 设计动机与交互原则（v2.3 修订）
原设置页以「文本框手输床号串」为编辑主交互（如 `309W01 309W02 309W03`），易输错、难校对、与「床号已结构化解析」的事实脱节。改进后：
1. **数据即输入**：床号已由 4.9.2 解析出 `ward / bedBase / bedType`，设置顺序应直接复用这些字段，而非让用户重新拼字符串。
2. **范围即顺序**：查房逐房进行，病房内床天然按基础床号升序；用「病区 + 床号范围（01–40）」即可表达整段顺序，无需逐床录入。
3. **选择优于键入**：真实加床从已识别清单勾选，归属病房从下拉选择，最大限度减少手动输入。
4. **结构化为主、文本为辅**：保留文本批量入口满足特殊场景，但降级为高级选项。

**与病人列表的关系**：病人列表（4.1.1）排序以本序列的 `roundingOrder` 为准；未进入序列的病人按「病区 + 基础床号数值」追加到末尾。

#### 4.9.4 数据模型字段

床号识别结果需持久化到数据模型，供排序与展示使用（完整 Schema 见第 7 章）：

- **Patient** 新增字段（见 7.1）：
  ```typescript
  ward: string      // 病区，如 "309W"（由 4.9.2 识别得出）
  room: string      // 病房标识，如 "309W41-43"（同房内床共享，由查房序列单元归属得出）
  bedBase: number   // 基础床号数值，如 23（由 4.9.2 解析）
  bedType: 'real' | 'extra-real' | 'virtual'  // 真实床 / 真实加床 / 虚拟床
  ```
- **Settings.roundingOrder**：存查房序列的单元数组，每个单元：
  ```typescript
  type Unit =
    | { kind: 'room'; ward: string; beds: string[] }        // 病房块，含房内真实床（及真实加床）
    | { kind: 'extra-real'; bed: string; room: string }     // 真实加床，指定归属病房
  ```

## 6. 视觉设计系统

### 6.1 设计原则

1. **温暖专业** - 使用暖色调（陶土色、米白色）营造医疗场景的信任感
2. **高信息密度** - 紧凑布局，一页展示尽可能多的信息
3. **清晰层级** - 通过颜色、字重、阴影区分信息优先级
4. **柔和动效** - 动画用于引导注意力，而非装饰

### 6.2 颜色系统

interface Patient {
  id: string           // 唯一标识
  bedNumber: string    // 床号
  name: string         // 姓名
  diagnosis: string    // 诊断
  group?: string       // 分组名称

## 9. 动画与动效

### 9.1 动效原则

1. **有意义** - 动画用于引导注意力或反馈操作结果
2. **克制** - 避免过多装饰性动画
3. **快速** - 大多数动画 200-300ms
4. **流畅** - 60fps，使用 transform 和 opacity
5. **可禁用** - 尊重 `prefers-reduced-motion` 设置

### 9.2 页面过渡

- Tab 切换：无过渡，直接切换
- 详情页进入：从底部滑入（300ms，ease-out）
- 详情页返回：向下滑出（200ms，ease-in）
- 设置页进入：从右侧滑入
- 实现说明：上述 route 级转场需基于 View Transitions API 或 Framer Motion 的 `AnimatePresence` 封装页面容器实现（Next.js App Router + Static Export 原生不支持页面级动画）。

### 9.3 组件动效

**列表项**：
- 进入：stagger slide-up，间隔 30ms
- 删除：向左滑出 + 高度收缩
- 新增：scale-in

**Bottom Sheet**：
- 打开：遮罩 fade-in，sheet slide-up（300ms）
- 关闭：sheet slide-down，遮罩 fade-out（200ms）

**Header 操作按钮**：
- 点击：缩放 0.97（轻量反馈），无展开菜单动画

**Toast**：
- 进入：从顶部/底部 slide-in（200ms）
- 退出：fade-out（150ms）
- 自动消失：3 秒

**Dialog**：
- 打开：scale-in + fade-in
- 关闭：fade-out

### 9.4 手势反馈

- 点击按钮：缩放 0.97
- 长按卡片：轻微放大 + 震动
- 左滑待办：阻力感，超过阈值后触发
- 下拉刷新：弹性回弹

### 9.5 性能要求

- 所有动画使用 transform/opacity
- 避免布局属性动画
- 列表滚动使用虚拟化（如病人数 > 100）
- 动画帧率 >= 60fps

## 10. 通知与提醒

### 10.1 提醒触发规则

**启动检查**：
- 每次打开 App 时检查所有待办
- 找出今天到期和已逾期的待办

**定时检查**：

**换药/查血提醒**：
- 根据 `lastDressingChange` 和 `dressingFrequency` 计算下次换药日期

**位置**：首页顶部
**显示条件**：
- 存在今天到期或已逾期的待办
- 存在需要换药/查血的病人

**样式**：
- danger：红色背景，"X 个待办已逾期 / Y 人需换药"
- warning：橙色背景，"今天有 X 个待办到期 / Y 人需查血"

### 10.3 浏览器通知

**触发条件**：
- 用户授权通知权限
- 定时检查时发现新的到期待办
- 同一待办不重复通知

**通知内容**：
- 标题："临床助手"

### 10.4 静默规则

- 用户未授权时不弹出系统提示
- 已完成待办不提醒
- 通知发送后 1 小时内不重复

## 14. 边界情况与异常处理

### 14.1 数据边界

- 病人数上限：10,000（设计上限，未压测）
- 单个病人待办上限：1,000（设计上限，未压测）
- 待办内容长度上限：200 字符
- 分组名上限：20 字符
- 超过上限时提示用户清理数据

### 14.2 交互边界

- 删除病人时：如果存在未完成待办，二次确认提示"该病人还有 X 个未完成待办"；确认删除后弹 Toast「已删除 · 撤销」，点击可恢复病人及其全部待办（约 5 秒窗口）
- 删除待办：立即从列表移除，同时弹 Toast「已删除 · 撤销」，点击撤销可恢复（约 5 秒窗口）
- 重复导入：按姓名更新，不重复创建
- 离线操作：正常保存，网络恢复后无需同步

### 14.3 解析边界

- 时间解析失败：允许保存，标记为"无日期"
- 多个时间：取优先级最高的第一个
- 过期日期：仍允许保存，显示"已逾期"
- 无法识别的日期：如"大后天"，显示"未识别"

### 14.4 导入边界

- 空文本导入：提示"请输入导入内容"
- 格式错误行（缺床号/姓名等）：跳过该行，并在预览中列出被跳过行号与原因，用户可修正后重新预览
- 重复姓名：更新信息，不报错
