### 1.2 潜在高风险点（基于 PRD / 架构的静态分析，标注【待实测】）

> 受沙箱限制，我**尚未在真实浏览器跑过应用**，以下为「潜在风险评估」，不是已确认的运行时缺陷。确认方案后执行 E2E/性能/a11y 会补齐「实测缺陷清单」。

1. **【待实测】动态路由直连访问**：`/patient/[id]` 是动态路由，纯静态导出下直接通过 URL 打开（非客户端导航）依赖 SW app-shell 兜底；SW 未注册或离线时可能白屏/404。
2. **【待实测】首屏性能**：静态导出 + Framer Motion + Dexie `useLiveQuery`，首屏 JS 体积与 TTI / LCP 待 Lighthouse 实测。
3. **【待实测】查房顺序拖拽持久化**：项目记忆里记录此处历史上踩过 Framer Motion `Reorder`「用索引作 value 导致回弹」的坑，是回归高风险点。
4. **【待实测】可访问性**：移动端 PWA 尚未做过自动 a11y 扫描，按钮/表单 `aria-label`、颜色对比度、键盘可达性待查（WCAG 2.1 AA）。
5. **【待实测】空 / 加载 / 错误态**：PRD 第 8 章要求三类状态，现有测试未覆盖 UI 层。
6. **【待实测】批量导入边界**：单测已覆盖部分场景，超长文本 / 特殊字符 / 空行等边界是否全盖待核。
7. **【待实测】PWA 离线**：SW precache + runtime 缓存在弱网 / 离线下行为待验证。
8. **【待实测】IndexedDB 数据隔离**：测试数据清理、跨会话持久化是否符合预期。

### 1.3 明确说明

- 上面 1.2 是「基于静态分析的潜在风险」，不是已确认的 bug。
- 现有 59 个单测**全过**，说明核心纯逻辑（解析/排序/导入导出）目前是稳的；但 UI 层、跨页流程、性能、a11y 仍是盲区。

## 2. 测试方案

### 2.1 整体策略

- **测试金字塔**：单元（70%）+ 适量集成/E2E（少量）+ 已有单测基线。
- **关键路径优先**：病人管理 → 待办 → 查房顺序 → 设置 → 通用待办/每日小结 → PWA 离线。
- **选择器原则**：`data-testid` > 语义角色 > 文本内容 > CSS class（最后选）。
- **每个测试独立**：不依赖执行顺序，浏览器 context 隔离 IndexedDB，避免数据污染。

### 2.2 分层计划

| 层 | 目标 | 方法 / 工具 | 产物 |
| --- | --- | --- | --- |
| 单元基线 | 核心逻辑回归 | vitest（**已完成，59 passed**） | 报告 |
| E2E | 关键用户旅程端到端 | Playwright + 本地 `out/`（等价线上） | spec + 运行结果 |
| 可访问性 | WCAG 2.1 AA 自动检测 | axe-core / playwright | 违规清单（含严重度） |
| 性能 | 首屏 / 体积 / CWV | Lighthouse（本地浏览器）或手动核查 | 评分 + 优化建议 |
| 视觉回归 | 关键页跨视口截图 | Playwright 截图（390×844、1280×800） | 截图基线 |

### 2.3 关键用户旅程（E2E 覆盖清单草稿）

1. 首页空状态 → 添加病人（表单校验：床号/姓名必填、时间解析） → 列表出现卡片
2. 病人详情：查看、编辑、删除、换药切换、待办增删改
3. 批量导入文本 → 解析为多病人 → 顺序正确
4. 待办：新增（含自然语言时间解析，如「明早查房」）→ 完成（左滑/勾选）→ 删除 → 通用待办分段
5. 查房顺序：拖拽重排 → 刷新后持久化
6. 设置：查房顺序编辑 / 床号识别模板 / 快捷待办 / 分组 各子页可打开并保存
7. 每日小结：生成与导出
8. PWA：离线后（断网）首页/已访问页仍可加载
